AGENTS.md guide

Source: AGENTS.md guide.html

Links

  • Standard
    • https://agents.md/
    • https://github.com/agentsmd

Practical conclusion

What AGENTS.md is for

AGENTS.md is a repo-local instruction file for coding agents. Its job is to give an agent the practical context it needs to work in a specific repository: commands, repo layout, validation rules, conventions, safety boundaries, and places to look before making larger changes. OpenAI Codex officially reads AGENTS.md before work and layers global, root, and nested project instructions. GitHub Copilot also supports AGENTS.md as agent instructions, with nearest-file precedence. Cursor documents AGENTS.md as a simple markdown alternative to .cursor/rules.

What it should not be used for

It should not be a second README, a full architecture manual, a task prompt, a style manifesto, or a pile of generic advice. Treat it as guidance, not enforcement. Claude Code’s docs are explicit that memory files are context, not enforced configuration, and that concise, specific instructions are followed more consistently.

Main recommendation

  1. Put exact commands near the top: install, test, targeted test, lint, format, type-check, build.
  2. Keep it short and operational. A practical target is under 150 to 200 lines, but the exact number is less important than removing low-value text.
  3. Include a definition of done: what checks must run before the agent claims success.
  4. Add safety boundaries: allowed freely, ask first, never do.
  5. Include only repo-specific conventions the agent cannot reliably infer from nearby code.
  6. Use nested AGENTS.md files only when subtrees genuinely differ, such as monorepos, frontend/backend splits, generated code, or infra folders.
  7. Do not trust AGENTS.md as a control mechanism. Back it with CI, tests, permissions, review, sandboxing, and explicit approval settings.
  8. Do not blindly generate a giant repo summary. The strongest research warning is that context files can add unnecessary requirements, increase cost, and sometimes reduce task success.
The Gemini and Claude outputs both converge on commands, concise instructions, safety boundaries, and avoiding generic bloat. I am not carrying forward their stronger claims about “universal” support, every-tool loading behavior, or unverified benchmark percentages unless backed by the web sources above.

Recommended AGENTS.md design rules

What to put near the top

Put operational content first.
  1. Commands
    • Install/bootstrap.
    • Run app locally.
    • Run targeted test.
    • Run full test suite.
    • Lint.
    • Format.
    • Type-check.
    • Build.
    • Regenerate artifacts, if relevant.
  2. Validation before completion
    • Required checks by change type.
    • Exact command-reporting rule.
    • What to say if a command cannot be run.
    • Whether tests/docs must be updated.
  3. Safety and approval boundaries
    • Allowed freely.
    • Ask first.
    • Never do.
  4. Repo layout
    • Only the important paths.
    • Entry points.
    • Tests.
    • Config.
    • Generated/vendor/infra folders.
Real public AGENTS.md files show this pattern in practice. OpenAI’s Apps SDK UI file lists setup plus required format, lint, type, and test commands. Prebid’s file includes programmatic checks, slow-test warnings, generated-file warnings, and affected-test guidance. OpenHands uses path-specific command and architecture guidance for backend, frontend, VS Code extension, enterprise code, and tests.

What to include only when relevant

Include these only when they solve real ambiguity:
  • Runtime versions, if the repo has strict requirements.
  • Monorepo/package map, if commands differ by package.
  • Architecture boundaries, if the agent could easily cross layers incorrectly.
  • Generated file rules, if generated files exist.
  • Database/migration rules, if schema changes are risky.
  • Security-sensitive rules, for auth, permissions, payments, secrets, infra.
  • Canonical code snippets, if short examples prevent repeated mistakes.
  • MCP/tool instructions, if the agent should use external docs, tickets, logs, or APIs.
  • Nested AGENTS.md files, if subtrees have materially different rules.

What to avoid

Avoid:
  • Generic advice: “write clean code,” “follow best practices,” “be careful.”
  • Full architecture essays.
  • Full API documentation.
  • README duplication.
  • Long dependency inventories.
  • Stale task-specific instructions.
  • Personal workflow preferences that belong in user-level instructions.
  • Secrets, tokens, credentials, private URLs, or production-only details.
  • Instructions that conflict with CI or package scripts.
  • Exact benchmark claims unless you can cite the actual study and caveat it.

Anti-patterns that make agents worse

Anti-pattern
Why it hurts
Huge auto-generated repo summaries
They add noise, cost, and unnecessary obligations.
Long lists of prohibitions
The agent may become conservative or over-explore. Pair “don’t” with “do this instead.”
Unverified commands
The agent wastes time or falsely reports validation.
Duplicated instruction files
Drift risk. One file says one thing, another says the opposite.

Agents can ignore or misinterpret instructions.

- Main stack: <language/runtime/framework/package manager>
- Primary code: `<path>`
- Tests: `<path>`

Run from the repository root unless noted.

- Run full test suite: `<command>`
- Lint: `<command>`

Before saying the task is complete:

1. Run the smallest relevant test or check for the changed area.
2. Run lint/type-check/build when relevant.
3. Report the exact commands run and whether they passed.
4. If a check cannot run, say why instead of claiming success.

Allowed without asking:

- Read and search files.

- Delete files.
- Run migrations.
- Change CI, release, deployment, or infrastructure config.
- Commit, push, rebase, reset, or force-push.

- Modify secrets, credentials, private keys, or real `.env` files.
- Use `--no-verify` unless explicitly approved.
- Silence failing tests or lint rules without explaining the root cause.

Notes on when to use each

Use the minimal version first. Add the complete sections only when the repo has real complexity or when you have evidence that agents repeatedly fail in a specific way.
Do not block full test/build commands by default. In some repos the full build is cheap and should be expected. In others it is slow, flaky, paid, or environment-dependent and should be “ask first.” Decide from the repo, not from a generic template.

Repo inspection checklist

Build/test/lint commands

Answer these from actual files, not guesses:
  • What installs dependencies?
  • What starts the dev server?
  • What builds the project?
  • What runs a single targeted test?
  • What runs the full test suite?
  • What runs lint?
  • What runs formatting?
  • What runs type checking?
  • What regenerates generated code or schemas?
  • Which commands need repo root?
  • Which commands need a package subdirectory?
  • Which commands are slow?
  • Which commands require Docker, services, network, credentials, or local databases?
  • What does CI run?
Inspect:
  • README.md
  • CONTRIBUTING.md
  • package.json
  • pnpm-workspace.yaml
  • turbo.json
  • pyproject.toml
  • requirements*.txt
  • Makefile
  • justfile
  • go.mod
  • Cargo.toml
  • Gemfile
  • Dockerfile
  • docker-compose.yml
  • .github/workflows/*
  • test/lint/type-check config files

Safety and approval boundaries

Decide:
  • Can the agent install packages?
  • Can it update lockfiles?
  • Can it delete files?
  • Can it run networked commands?
  • Can it run migrations?
  • Can it change CI?
  • Can it touch infra?
  • Can it commit?
  • Can it push?
  • Can it publish/deploy?
  • Can it edit generated files?
  • Can it modify config outside the repo?
Good default:
  • Allowed: read/search files, edit task-related files, run local checks.
  • Ask first: dependency changes, destructive commands, networked shell commands, migrations, CI/infra/release changes, commits/pushes.
  • Never: secrets, production credentials, --no-verify without approval, hiding failures.

Files/folders to avoid

Common candidates:
  • .env
  • .env.* except safe examples like .env.example
  • credentials, private keys, certificates
  • node_modules/
  • vendor/
  • generated clients
  • build outputs
  • coverage outputs
  • migration directories, unless task requires them
  • production deployment config
  • large snapshots or fixtures
  • lockfiles, unless dependency changes require them

Docs to read before larger changes

List only useful docs:
  • README.md
  • CONTRIBUTING.md
  • docs/ARCHITECTURE.md
  • docs/TESTING.md
  • docs/DEVELOPMENT.md
  • docs/RELEASE.md
  • docs/SECURITY.md
  • ADRs
  • package-specific READMEs
  • API/schema docs
  • design-system docs

Validation rules

Define checks by change type:
  • Docs-only change: markdown lint or no code checks, depending on repo.
  • Frontend change: targeted test, lint, type-check, maybe build.
  • Backend change: targeted test, lint, type-check, maybe integration test.
  • CLI/API behavior change: tests plus docs.
  • Generated code change: regenerate command plus check generated diff.
  • Dependency change: install, lockfile update, tests, security/license consideration.
  • Infra change: plan/check only, never apply without approval.
Require the agent to say:
  • Exact commands run.
  • Pass/fail result.
  • Why any check was skipped.
  • Whether tests/docs were updated.

Whether nested AGENTS.md files are useful

Use nested files when:
  • A monorepo has different package managers.
  • Frontend and backend have different test commands.
  • Infra has stricter safety rules.
  • Generated-code directories have special rules.
  • A subtree has a different language/runtime.
  • A package has its own CI or release workflow.
Avoid nested files when:
  • The root file can describe the difference in a few bullets.
  • The nested file would mostly duplicate the root.
  • The purpose is just to hide root-file bloat.
  • Tool support for nested files is unclear in your target agent.

Conversion-ready prompt

First Pass

You are helping generate an AGENTS.md file for this repository.

Important:
Do not write or modify AGENTS.md yet unless I explicitly ask you to edit the file.
First inspect the repo, then show me the proposed AGENTS.md content and wait for approval.

Goal:
Create a concise, practical operating guide for coding agents working in this repo.
This is not a README, architecture manual, task prompt, or full repo summary.
It should contain only the information that helps an agent avoid mistakes, run the right checks, and respect safety boundaries.

Before drafting, inspect the repository:

1. List the repository root.
2. Read existing instruction files if present:
- AGENTS.md
- CLAUDE.md
- GEMINI.md
- .github/copilot-instructions.md
- .github/instructions/**/*.instructions.md
- .cursor/rules/*
- .cursorrules
- other agent or AI instruction files
3. Read project overview docs:
- README.md
- CONTRIBUTING.md
- docs/ files that look relevant
4. Inspect build and package files:
- package.json
- pnpm-workspace.yaml
- turbo.json
- pyproject.toml
- requirements*.txt
- Makefile
- justfile
- go.mod
- Cargo.toml
- Gemfile
- Dockerfile
- docker-compose.yml
- equivalent files for this repo
5. Inspect CI:
- .github/workflows/*
- other CI config
6. Inspect key source and test directories enough to understand layout and test conventions.

Hard rules:
- Do not invent commands.
- Do not invent conventions.
- Do not invent runtime versions.
- Do not include generic advice like “write clean code,” “follow best practices,” or “be careful.”
- Do not duplicate README or architecture docs.
- Do not write a full repo map.
- Do not include long explanations.
- Do not include secrets, private URLs, credentials, or machine-local assumptions.
- Treat AGENTS.md as guidance, not enforcement.
- Prefer TODO placeholders over guessing.
- If a command cannot be inferred, write prose such as:
“No dedicated format command found. Do not invent one.”
Do not put TODOs inside shell code blocks.
- Do not list risky write/apply/publish/deploy/destructive commands as normal runnable commands.
Mention them only under approval boundaries.
- Do not bundle commands with different approval levels in the same code block.

Length and pruning rules:
- Target 400 to 800 tokens for a normal single-stack repo.
- Hard maximum: 1,000 tokens unless the repo has multiple stacks, separate apps, or genuinely different validation pipelines.
- Do not target a specific line count. Lines are a ceiling, not a goal.
- Shorter is better if the repo has one stack and one validation path.
- Include only details that would change agent behavior.
- Omit repo-map detail unless it helps navigation or prevents edits in the wrong place.
- Omit individual test-file mappings unless they clearly prevent likely mistakes.
- After drafting, remove any line that is merely descriptive, obvious from the repo tree, duplicated in README/docs, or unlikely to prevent a mistake.

Command classification rules:
Classify commands by risk before placing them in the file.

Safe validation/development commands:
- tests
- targeted tests
- lint
- type-check
- build
- local read-only inspection
- non-destructive local dev commands

These may go in the Commands section.

Write-producing but normal commands:
- code generation
- dashboard/report generation
- clean commands that delete generated output
- local artifact regeneration

Include these only if commonly needed for this repo.
Label when to run them, for example:
“Run only when generated output is part of the requested task.”
or
“Run only when cleaning build output is needed.”

Network/dependency commands:
- install commands
- package manager update commands
- tool installation commands
- dependency sync commands that may use the network

Include these only as setup commands and clearly mark “ask first.”
Do not bundle them with safe commands like build or test.

Risky write/apply commands:
- deploy
- publish
- release
- terraform apply or equivalent
- database migration apply
- policy apply with --yes or equivalent
- force reset
- destructive cleanup
- production writes

Do not include these as runnable commands in the Commands section.
Mention them only under approval boundaries.

Good examples:
- module boundaries
- generated config shape restrictions
- import rules
- data-access patterns
- compatibility rules
- deterministic output requirements

Compress related rules.
Use short examples only when they prevent likely mistakes.
Prefer nested bullets for long inline lists of config shapes, file mappings, or examples.

Do not include architecture material that belongs only in docs and is not needed during routine coding.

## Docs
List only docs agents should read before larger changes.

Recommend nested files only if subtrees have genuinely different commands, stacks, ownership, or safety rules.
Do not recommend nested files just to organize a long root file.

Before showing the proposed AGENTS.md, run a final boundary-conflict review.

Check for and fix:
- Any command listed in normal Commands that is also described as ask-first or never.
- Any command block that bundles safe commands with networked, destructive, write-producing, or approval-required commands.
- Any file/folder described as editable but later marked generated, vendor, build output, secret-bearing, or off-limits.
- Any write-producing command that lacks a “run only when needed/requested” qualifier.
- Any generated/vendor/build output that is mentioned without clarifying whether it may be read, edited, or regenerated.
- Any git state-changing command, including commit, push, rebase, reset, force-push, checkout that discards work, or --no-verify. Put these under “Never unless explicitly approved,” not normal workflow.
- Any risky apply/deploy/publish/release/migration command. Put these only under approval boundaries, not Commands.
- Any duplicate safety rule. If one “Never unless explicitly approved” bullet already covers a narrower dangerous case, merge them.
- Any confusing safe-template wording. Do not say “never treat .env.example as secret-bearing.” Instead say safe templates may be edited only if they do not contain real secrets.
- Any long inline list of config shapes, file mappings, or examples that would be clearer as nested bullets.

After the review, revise the draft before presenting it.

1. Show the proposed AGENTS.md in a markdown code block.
2. List 3 to 7 things you intentionally left out to keep it concise.
3. List unresolved TODOs or questions.
4. End with exactly one simple approval question:
“Do you want me to apply this AGENTS.md to the repo?”

Do not write or modify the file unless I explicitly approve.
Do not ask multiple questions at the end.

Important:
- Do not write or modify the file unless I explicitly ask.
- Do not add new repo facts unless they are already present in the draft or verified from files already inspected.
- Do not invent commands, conventions, runtime versions, or safety rules.
- Do not return the same file unless you can explicitly show that every checklist category below is already clean.
- If you find any checklist issue, produce a revised AGENTS.md.

General length target:
- Target 400 to 800 tokens for a normal single-stack repo.
- Hard maximum: 1,000 tokens unless the repo has multiple stacks, separate apps, or genuinely different validation pipelines.
- If the draft is over 1,000 tokens and the repo does not clearly justify it, treat overlength as a material issue.
- Prefer deleting low-value sections over polishing them.
- Keep only details that would change agent behavior.

Required cleanup checklist:

1. Boundary conflicts

Check for:
- Commands listed as normal workflow but also marked ask-first or never.
- Command blocks that bundle safe commands with networked, destructive, write-producing, or approval-required commands.
- Files/folders described as editable but later marked generated, vendor, build output, secret-bearing, or off-limits.
- Write-producing commands that lack “run only when needed/requested” qualifiers.
- Generated/vendor/build output mentioned without clarifying whether it may be read, edited, or regenerated.
- Git state-changing commands outside “Never unless explicitly approved.”
- Risky apply/deploy/publish/release/migration commands outside approval boundaries.

2. Command safety and usefulness

Check that:
- Install/dependency commands are separated from build/test commands.
- Build/type-check/test/lint commands are safe validation commands unless the repo shows otherwise.
- Clean/codegen/dashboard/report commands are clearly labeled as write-producing if applicable.
- Risky write/apply commands are not shown as normal runnable commands.
- Missing commands are described in prose, not fake shell TODO blocks.
- Commands are exact and grounded in inspected repo files.
- Git inspection commands use non-interactive forms when useful, for example `git --no-pager status --short` instead of commands that may open a pager.

3. Safety wording

Check that:
- “Allowed without asking,” “Ask first,” and “Never unless explicitly approved” are distinct.
- Secret-bearing files are handled clearly.
- Generated/build/vendor outputs are read-only for context unless explicitly requested.
- Commits, pushes, force-pushes, rebases, resets, destructive checkouts, and --no-verify are under “Never unless explicitly approved.”
- Exceptions to safety rules are attached to the relevant rule, not placed as standalone bullets.
- Every safety bullet makes sense on its own without relying on the section heading to reverse or clarify its meaning.

Secret/template rule:
- Do not create a standalone bullet about safe templates, example files, sample env files, generated samples, or placeholders under “Never unless explicitly approved.”
- Do not write anything like “treat safe templates as secret-bearing.”
- Correct pattern:
“Do not modify secrets, credentials, private keys, real environment files, or credential files. Safe templates or examples may be edited only if they do not contain real secrets.”
- Keep exact file examples from the existing draft if they are useful, but attach the safe-template exception to the same bullet as the secret-files rule.

4. Missing repo-specific invariants

Check whether the draft preserves the most important repo-specific rules discovered during inspection.

Look especially for:
- data contracts
- schema or artifact compatibility rules
- versioning rules
- generated-file ownership rules
- benchmark or measurement semantics
- migration or deployment boundaries
- security or privacy assumptions
- compatibility guarantees
- deterministic output requirements
- domain-specific correctness rules
- test assertions that encode important project policy

If a missing invariant is likely to prevent a wrong edit, add it concisely under “Architecture or code conventions,” “Testing conventions,” or “Safety and approval boundaries.”

Do not add broad architecture explanation. Add only short operational rules that would change agent behavior.

5. Pruning

Remove or compress:
- Generic advice.
- README duplication.
- Full repo-map detail.
- Long architecture explanation.
- Repeated safety rules.
- Obvious facts the agent can infer from the repo tree.
- Stale-prone observations that do not help day-to-day agent behavior.
- Anything unlikely to change agent behavior.
- Exhaustive test-file mappings unless they clearly prevent likely mistakes.

Keep:
- Useful repo-specific rules.
- Exact validation commands.
- Safety boundaries.
- Short repo navigation hints.
- Known failure modes that are likely to prevent wrong edits.
- Missing repo-specific invariants that would prevent wrong edits.

6. Scanability

Improve:
- Convert bullets with three or more inline examples, commands, file mappings, config shapes, forbidden patterns, or allowed patterns into nested bullets by default.
- Keep them inline only if each item is very short and the full bullet remains easier to scan than a nested list.
- If a bullet introduces allowed shapes, forbidden shapes, command examples, file mappings, or config examples, prefer this structure:

- Main rule:
- `example one`
- `example two`
- `example three`

- Merge repeated similar rules.
- Fix awkward wording.
- Keep section headings consistent.
- Make each command description’s risk level clear.

Do not over-format ordinary prose lists. This rule is mainly for examples, patterns, commands, mappings, and config shapes.

7. Accuracy

Flag, but do not silently fix:
- Commands that seem unverified.
- Runtime versions that may be inferred but not proven.
- Claims about CI when CI files were not inspected.
- Conventions that sound plausible but are not grounded in repo files.
- TODOs that should remain TODOs instead of being guessed.

For stale-prone CI wording:
- Prefer conditional wording such as “If no repo-local CI workflow exists...” over absolute wording such as “No CI exists...” unless the task explicitly asks for a snapshot statement.

8. Minimal-change discipline

Prefer small, targeted edits over rewriting the whole file, except when the draft exceeds the length target or misses important repo-specific invariants.

Preserve the existing structure unless:
- the file is over the length target,
- safety boundaries are unclear,
- scanability is poor,
- or a structure change is needed to add or preserve important repo-specific invariants.

Do not over-optimize wording that is already clear.

Output format:

1. Verdict:
- Say whether the current AGENTS.md is usable as-is, usable with edits, or not ready.

2. Material issues found:
- List only issues that matter.
- For each issue, explain why it matters.

3. Revised AGENTS.md:
- Show the full revised file in a markdown code block.
- The revised file must not be identical to the input if any checklist issue exists.

4. Remaining questions/TODOs:
- List only items that need human confirmation.
- If none remain, say “None.”

5. Approval handoff:
- End with exactly one simple question:
“Do you want me to apply this revised AGENTS.md to the repo?”

Do not write or modify the file unless I explicitly approve.
Do not ask multiple questions at the end.

Best default AGENTS.md policy

  • One root AGENTS.md by default.
  • Put exact commands near the top.
  • Prefer targeted validation before full-suite validation.
  • Require agents to report commands run and results.
  • Keep the file short. Cut anything obvious or rarely relevant.
  • Treat it as guidance, not enforcement.
  • Use safety tiers: allowed, ask first, never.
  • Do not include secrets, credentials, or production-only details.
  • Do not duplicate README or architecture docs.
  • Add nested files only for genuinely different subtrees.
  • Let the file grow from repeated agent mistakes, not from speculative completeness.
  • Have AI draft it only after repo inspection, then review before writing.
